<objective>
Create the database schema for normative ranges and build the query layer that lets the rating engine use DB-backed ranges with hardcoded fallback.

Purpose: This is the foundation for all admin panel functionality. Without DB tables and a query layer, no CRUD UI or versioning can work. The rating engine must seamlessly prefer DB overrides while falling back to hardcoded data.ts values.

Output: Two new DB tables (normative_ranges, normative_versions), migration SQL in both PG and SQLite branches, a db-ranges.ts query module, and a refactored ratings.ts that supports pre-loaded DB ranges.
</objective>

**getDbStandards(testKey, gender?, ageGroup?):** Query normativeRanges table for a single marker variant. Build conditions array: always include `eq(schema.normativeRanges.testKey, testKey)`. If gender is provided, add `eq(schema.normativeRanges.gender, gender)`. If ageGroup is provided, add `eq(schema.normativeRanges.ageGroup, ageGroup)`. If neither gender nor ageGroup, add `isNull` conditions for both. Use `db.select().from(schema.normativeRanges).where(and(...conditions)).limit(1)`. Return `{ unit, note, tiers, severityWeight }` or null if no rows.

**getAllDbRanges():** `db.select().from(schema.normativeRanges)` -- returns all rows. Used by admin browse page and version snapshot creation.

**getDbRangesByTestKey(testKey):** `db.select().from(schema.normativeRanges).where(eq(schema.normativeRanges.testKey, testKey))` -- returns all variants for one marker.

**upsertDbRange(data: { testKey, category, gender?, ageGroup?, unit?, note?, tiers: TierRanges, severityWeight?: number }):** Check if row exists with same testKey+gender+ageGroup. If exists, UPDATE tiers, unit, note, severityWeight, updatedAt. If not, INSERT new row with createdAt and updatedAt set to `new Date().toISOString()`. Return the upserted row.

**deleteDbRange(testKey, gender?, ageGroup?):** DELETE from normativeRanges where testKey matches (and gender/ageGroup if provided). Return number of deleted rows.

2. Add `preloadDbRanges()` function to `src/lib/normative/db-ranges.ts`:
This loads ALL normativeRanges rows and builds a lookup map: `Record<string, { unit, note, tiers, severityWeight, gender?, ageGroup? }[]>` keyed by testKey. This is called ONCE before report generation to avoid N+1 queries.

3. Refactor `src/lib/normative/ratings.ts`:

Keep the existing `getStandards()` function UNCHANGED (it remains the synchronous hardcoded-only path).

Add a NEW exported function `getStandardsWithOverrides(testKey, age, gender, dbRangesMap)`:
- `dbRangesMap` is the Map returned by `preloadDbRanges()`
- First, check if `dbRangesMap` has entries for `testKey`
- If yes, find the best match: exact gender+ageGroup match > gender-only match > unisex match
- Return `{ unit, note, standards: tiers, severityWeight }` from the DB row
- If no DB match found, fall back to `getStandards(testKey, age, gender)` (existing hardcoded logic)

Add `getStandardsFromSnapshot(snapshot: NormativeVersionSnapshot, testKey, age, gender)`:
- Import `NormativeVersionSnapshot` from `@/types/normative`
- `snapshot` is the parsed JSON from `normative_versions.ranges_json`, typed as `NormativeVersionSnapshot`
- Look up `testKey` in snapshot to get a `NormativeVersionMarker`
- From the marker's `variants` array, find the best match: exact gender+ageGroup > gender-only > first variant (unisex fallback)
- If found, return `{ unit: marker.unit, note: marker.note, standards: variant.tiers }`
- If not found in snapshot, fall back to `getStandards(testKey, age, gender)`

Do NOT change the signature or behavior of `getPeak360Rating()` -- it remains synchronous and uses hardcoded data. Report generation (Section 11) will use the new `getStandardsWithOverrides` or `getStandardsFromSnapshot` functions directly.
