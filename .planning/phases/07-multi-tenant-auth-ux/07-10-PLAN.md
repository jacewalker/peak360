<objective>
Close the two security blockers in Better Auth configuration and the admin role-change handler.

**BL-01 — VERIFICATION.md row 2 ("minPasswordLength enforced at the API layer matches the UI's 8-character claim"):**
src/lib/auth.ts line 17 reads `minPasswordLength: 4`. The /reset-password UI's `<input minLength={8}>` and the milestone constraint that "blood results and medical screening data require encryption at rest" require an 8-character floor. The fix is a one-line change: 4 to 8.

**BL-02 — VERIFICATION.md row 4 ("Last-admin guard rollback is reachable and correctly restores role on concurrent race"):**
src/app/api/admin/users/[userId]/role/route.ts lines 71-109 implement a post-check rollback that re-counts admins after auth.api.setRole and tries to call auth.api.setRole again to restore the demoted user. In the canonical race scenario (two admins demote each other concurrently), the caller's session is no longer admin by the time the rollback runs, so the rollback setRole enters the catch{} block silently — the user stays demoted while the response claims "Previous role restored." Replace the count + setRole + post-check + rollback design with an atomic Drizzle transaction wrapping the count + role write directly, so the race cannot occur. auth.api.setRole is called once OUTSIDE the transaction (post-commit) only for Better Auth's session-invalidation side-effects, NOT for the role write itself — which means we are no longer dependent on the caller's session still being admin at the moment of the role write.

The two existing regression tests (tests/security/auth-config.test.ts and tests/security/last-admin-guard.test.ts) need targeted edits so they continue to validate the new code shape.

Output: 2 source files modified + 2 test files updated.
</objective>
