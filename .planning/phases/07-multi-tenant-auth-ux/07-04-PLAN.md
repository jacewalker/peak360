<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| anonymous browser → /login | Untrusted form input (email, password, magic-link request) — Better Auth handles validation server-side. |
| authClient.forgetPassword → Better Auth | Email enumeration risk; mitigated by Better Auth's non-disclosure semantics on the forget-password endpoint. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-13 | Elevation of Privilege | Removed "Create one" link could be re-added in a future commit, restoring the public coach-signup attack surface | mitigate | Plan 07-01 also enforces `disableSignUp: true` in auth.ts (server-side). Even if a future commit accidentally re-renders a signup form, the Better Auth API rejects the request. Plan 07-09 contributes a regression test asserting `disableSignUp: true` is present. |
| T-07-14 | Information Disclosure | Forgot-password endpoint reveals whether an email is registered | accept | Better Auth's forget-password handler returns the same response shape regardless of user existence. The success banner copy is identical for known + unknown emails. |
| T-07-15 | Phishing | Magic-link delivered to an attacker-controlled inbox if the user mistypes their email | accept | Out of scope for this phase. Better Auth tokens have a 5-minute expiry; impact is bounded. |
| T-07-16 | Tampering | Mode toggle could be tampered with via DOM to bypass intended flow | accept | The toggle is purely cosmetic — both modes call the same Better Auth endpoints which validate server-side. No security boundary is at the toggle. |
</threat_model>
