# Quick 260525-ls8 — Modernize the Playwright e2e suite

## Why
The e2e specs predate the Phase 7 auth + `/portal` route migration and Phase 9 redesign.
They hit stale routes (`/assessment/...`, `/clients`, `/assessments`) and call
`POST /api/assessments` unauthenticated (now requires a session) → 32 failures that are
purely staleness, not regressions. Modernize them to exercise the current app.

## Environment for running the suite (IMPORTANT)
- This runs in a git worktree, which has **no `.env.local`** (it's gitignored). Sign-in needs the
  real dev DB (Mac mini Postgres) that has `admin@admin.com`. **First action in the worktree:**
  `cp /Users/jace/Code/peak360/.env.local ./.env.local` so `DATABASE_URL` is present.
  (The worktree has its own `.next`, so its `npm run dev` on :3000 will NOT clash with the
  user's :8080 dev server.)
- The test server must trust its own origin: set it in `playwright.config.ts` (see Task 4).
- Creds: `admin@admin.com` / `password123` (the seeded admin). Tests run against the **shared
  dev Postgres**, so every test MUST delete any assessment it creates (afterEach/afterAll cleanup).

## Current facts (verified)
- Routes: `/portal`, `/portal/clients`, `/portal/assessments`, `/portal/assessment/[id]/section/[num]`,
  `/portal/assessment/[id]/report`. Landing is `/` (public, behind a gate).
- `POST /api/assessments` (authed) accepts `{ clientName, clientEmail, clientDob, clientGender }`,
  returns `{ success, data: { id } }`. `PUT /api/assessments/[id]/sections/[num]` body `{ data: {...} }`.
  `DELETE /api/assessments/[id]`. All require a coach/admin session; clients get 403.
- The report lives at `/portal/assessment/[id]/report` (Section 11 content / pillars).

## Task 1 — Shared helpers (`e2e/helpers/`)
Create `e2e/helpers/auth.ts`:
- `signInAsAdmin(page)`: `await page.request.post('/api/auth/sign-in/email', { data: { email, password } })`,
  assert `res.ok()`. Creds from `process.env.E2E_ADMIN_EMAIL ?? 'admin@admin.com'` and
  `E2E_ADMIN_PASSWORD ?? 'password123'`.

Create `e2e/helpers/assessment.ts`:
- `createAssessment(page, { clientName, clientGender?, clientDob?, clientEmail? })` → POST, return `data.id`.
- `seedSection(page, id, num, data)` → PUT `/api/assessments/${id}/sections/${num}` `{ data }`.
- `deleteAssessment(page, id)` → DELETE (best-effort, ignore errors).
All use `page.request` so they share the signed-in cookie.

## Task 2 — Rewrite `phase1-report.spec.ts`

  (clientName, clientGender, clientDob/clientAge as the old spec did) and Section 5/biomarker data

- Navigate to `/portal/assessment/${id}/report` (NOT `/section/11`) for report assertions; use

Never hardcode an assessment id. Creds + base from env with sensible defaults.
