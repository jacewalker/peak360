<objective>
Wire the UI for admin impersonation. From the admin People page an admin can click
"Log in as" on any coach or client to switch into that user's session and see exactly
what they see; a persistent banner across the portal lets the admin switch back to
their own account.

Purpose: Lets admins debug and support coaches/clients by experiencing the app exactly
as that user does, without sharing passwords.

Output:
- A new client component `src/components/portal/ImpersonationBanner.tsx`
- The portal server layout renders the banner globally when impersonating
- The People page gains a "Log in as" action in both desktop and mobile views

IMPORTANT — this is a UI-wiring task ONLY. The backend already exists:
- Server: `admin()` plugin configured in `src/lib/auth.ts` (adminRoles: ['admin'])
- Client: `adminClient()` in `src/lib/auth-client.ts` exposing
  `authClient.admin.impersonateUser({ userId })` and `authClient.admin.stopImpersonating()`
- DB: `session.impersonatedBy` column already exists in both `schema.ts` and `schema-sqlite.ts`
DO NOT create or modify any auth endpoint, the DB schema, or the admin plugin config.
</objective>

Confirmed from node_modules/better-auth/dist/plugins/admin/admin.d.mts (do NOT re-verify):

- `authClient.admin.impersonateUser({ userId: string })` — POST /admin/impersonate-user.
  Body is exactly `{ userId }`. On success the server rotates the session cookie so
  subsequent requests resolve as the impersonated user. Better Auth client methods
  return `{ data, error }` (error is null on success). Treat a non-null `error` as failure.
- `authClient.admin.stopImpersonating()` — POST /admin/stop-impersonating. Takes NO
  arguments (server uses `requireHeaders: true`; no body). Same `{ data, error }` shape.

Call pattern (mirror the existing fetch error handling in this codebase):
  const { error } = await authClient.admin.impersonateUser({ userId: u.id });
  if (error) { /* show error Toast */ } else { router.push('/portal'); router.refresh(); }
</verified_signatures>
