# Quick 260331-r70: Fix PDF Page-Break Overflow — Summary

Three surgical fixes to Section11's `exportPdf` that eliminate content clipping at A4 page boundaries: scroll-to-top before DOM measurement, synchronous reflow after each spacer, and a 4mm bottom guard constant.

## What Was Done

### Task 1: Apply three targeted fixes to exportPdf

**Change 1 — Scroll stabilisation**

Added `window.scrollTo(0, 0)` immediately after grabbing the container reference, before any `getBoundingClientRect` calls. Without this, if the user had scrolled down, all top measurements were offset by `window.scrollY`, causing spacers to be inserted at wrong positions. The previous scroll position is captured and restored after spacers are removed.

**Change 2 — Synchronous reflow after each spacer**

Added `void container.offsetHeight` inside the spacer `forEach` immediately after `spacers.push(spacer)`. The `offsetHeight` property access forces the browser to flush all pending DOM mutations synchronously before the next `getBoundingClientRect` read. Without this, the browser may batch DOM updates and return stale positions for subsequent elements.

**Change 3 — PDF bottom guard constant**

Replaced the flat constants block with:
```typescript
const PDF_MARGIN = 6; // mm margin on each page
const PAGE_BOTTOM_GUARD = 4; // mm buffer against sub-pixel rounding drift across pages
const imgWidth = 210 - PDF_MARGIN * 2;
const pageHeight = 297;
const usableHeight = pageHeight - PDF_MARGIN * 2 - PAGE_BOTTOM_GUARD;
```

The 4mm guard absorbs accumulated sub-pixel rounding errors that cause the last pixel row of a page slice to overflow into the next page. `imgWidth` now subtracts margins so content is inset from the physical page edges.
