<role_detection_decision>
ROLE-DETECTION + WRITE-GATING MECHANISM (single source of truth):
- The section page (`section/[num]/page.tsx`) is a `'use client'` component. Read the role via `authClient.useSession()` — the SAME mechanism used by Sidebar.tsx and the dashboard (consistency).
- Define ONE derived flag: `const canWrite = role === 'coach' || role === 'admin';` (POSITIVE equality). This is the single source of truth for "is this viewer allowed to mutate the assessment".
- Fail-safe: while the session is loading, `role === undefined` ⇒ `canWrite === false` ⇒ no writes. This is correct: the existing save effects already gate on `store.isDirty && loaded`, and saving only happens AFTER a user edits a field. A coach cannot have edited before their session resolves (sub-second), so coach editing is NOT broken — writes simply (re-)enable the instant the session resolves, before any debounce can fire from a real edit.
- `READ-ONLY` for the UI = `!canWrite`. Treat unknown/loading conservatively (read-only) so a client never momentarily gets a writable form.

DEFAULT-TO-11 MECHANISM:
- Primary: client entry-point links route to `/section/11` instead of `/section/${currentSection}`. Gate these with positive `role === 'client'`.
  - Dashboard `AssessmentRow` (portal/page.tsx) and the action-item links — clients only reach assessments via these or the assessments list.
  - The assessments-list row click (assessments/page.tsx) for clients.
- Backstop: the index redirect page (`assessment/[id]/page.tsx`) sends clients to `/section/11`, others to `currentSection`. (This page currently redirects to the wrong `/assessment/...` path missing the `/portal` prefix — FIX it to `/portal/assessment/...` while here.)
- After landing on 11, the client can still use Prev/Next nav (NavigationButtons) to reach sections 1–10 read-only, so default-to-11 does not trap them.
- Coaches/admins keep landing on `currentSection` everywhere — only the `role === 'client'` branch changes the target.
</role_detection_decision>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser (client role) → section page | Client JS could attempt to mutate assessment data via store/save effects |
| browser → /portal/assessment/[id]/section/* (URL) | A client guessing/altering an assessment UUID attempts to view another client's data (IDOR) |
| browser → assessment write APIs (PUT/DELETE/sendBeacon) | Read-only enforcement could be bypassed if any write path remains wired for clients |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-i23-01 | Tampering | client write attempt via auto-save / sendBeacon / completion / status PUT | mitigate | Task 4 gates EVERY write path on `canWrite = role==='coach'||role==='admin'`; loading/unknown role ⇒ no writes (fail-safe). `handleChange` no-ops so the store never goes dirty for clients; auto-save + beacon effects early-return; `navigate` skips save. |
| T-i23-02 | Elevation/Information Disclosure | IDOR — client opens another client's assessment by URL | mitigate | Task 3 adds an SSR ownership gate in section/layout.tsx (fetch row + `hasAccess` + redirect/404 BEFORE children render), replacing the implicit gating the removed client→/report redirect provided. Does not rely on client-side fetch 403. |
| T-i23-03 | Tampering | read-only UI bypass (re-enabling a disabled input via devtools) | mitigate | Defense-in-depth: native `fieldset[disabled]` + `pointer-events-none` is the visible layer; the authoritative layer is T-i23-01 (handleChange no-op never updates the store) AND the server-side write authorization on the assessment PUT/DELETE/section routes (unchanged, still enforces role/ownership). | T-i23-SC | Tampering | npm/pip/cargo installs | mitigate | No new dependencies are installed in this plan; all imports (authClient, db, drizzle eq, AuthSession) already exist in the codebase. No package-legitimacy checkpoint required. |
