# Quick 260527-i23 Plan 01: Client Read-Only Assessment Summary

Clients now get an Assessments sidebar link and a read-only assessments list, land on the interactive Section 11 report when opening an assessment, can browse sections 1–10 fully read-only with zero writes, and are blocked from other clients' assessments by an SSR ownership gate — while coach/admin editing, auto-save, navigation, and completion remain byte-for-byte unchanged.

## What Was Built

- **Task 1 — Sidebar (`7cf0a24`):** Client role branch returns `[Dashboard, Assessments]`; Clients link stays coach/admin-only; loading fallback unchanged.
- **Task 2 — Assessments list (`9f76e69`):** Added `authClient.useSession()` + `canManage = role==='admin'||'coach'` (positive equality). Gated Start new, Export/Import CSV (both empty + populated toolbars), Select all + divider, Delete N selected, per-row checkbox, per-row Assign, per-row Delete. Client empty-state copy carries no create prompt ("No assessments have been shared with you yet."). Row click routes clients to `/section/11`, coach/admin to `currentSection`. List, search, metrics bar still visible to clients.
- **Task 3 — SSR gate + routing (`7ef2f74`):** Replaced the forced client→report redirect in `section/layout.tsx` with an SSR `hasAccess` ownership gate (require session → fetch row → `notFound()` on missing → `redirect('/portal')` for non-owners → owners render children). Index `assessment/[id]/page.tsx` now waits for session, routes clients to `/section/11` and others to `currentSection`, and fixes the legacy missing `/portal` prefix. Dashboard rows + action items route clients to `/section/11`. Gated the dashboard Clients counter tile behind coach/admin.
- **Task 4 — Read-only section page (`15f6165`):** Added `canWrite`/`readOnly` from `useSession()`. Every write path gated on `canWrite` (auto-save effect, beforeunload beacon, `saveSection`, `navigate`, `handleChange`). Sections 1–10 wrapped in a native disabled `<fieldset>`; Section 11 left fully interactive. NavigationButtons hides Complete/SaveExit/Cancel for read-only clients on both branches.

## Plan-Checker Fixes Applied

1. **Defense-in-depth write guards (Task 4):** `handleComplete` (status PUT), `handleSaveExit` (save), and `handleCancel` (DELETE) each have `if (!canWrite) return;` as the first line — so even if a button were somehow rendered for a client, no write/DELETE/status-PUT can fire. Combined with the UI removal of those entrypoints, the "zero client writes" guarantee is defense-in-depth, not UI-only. (7 `if (!canWrite) return;` guards present in the section page.)
2. **Dashboard Clients tile (Task 3):** The counter-strip "Clients" tile linking to `/portal/clients` is now gated behind `userRole === 'coach' || userRole === 'admin'` (positive check). The two Assessments tiles stay visible to clients.
